# v0.2.0 — Pure Plugin Design

**Status:** EXPERIMENTAL. Depends on an unmerged upstream RFC.
**Branch:** `feat/v0.2.0-pure-plugin`. Not for release until the host
ships the identity primitive described below.

This document describes the v0.2.0 architecture of
`claudecodeui-plugin-account`: a self-contained plugin that runs as a
Node.js subprocess under the cloudcli/claudecodeui plugin server
infrastructure, with **no host patches** and **no host UI integration
patches**. v0.1.x worked by patching the host (`server-patch/`,
`host-ui-integration/`); v0.2.0 replaces those patches with a pure
in-plugin implementation that depends only on a well-defined identity
contract from the host.

## Why a redesign

v0.1.x ships three artifacts:

1. `server-patch/` — patches `server/routes/auth.js` and
   `server/modules/database/repositories/users.ts` to add
   `PUT /api/auth/account/{password,username}` and rate-limiting.
2. `host-ui-integration/ui-integration.patch` — adds an "Account" tab
   to the host's Settings UI.
3. `src/` — a vanilla-DOM plugin frontend that calls
   `/api/auth/account/*`.

The patches are unavoidable in v0.1.x because the host's plugin
subprocess proxy strips `req.user` before reaching the plugin (see
[`docs/internal/cloudcli-plugin-system-analysis.md`](internal/cloudcli-plugin-system-analysis.md)
§3.1, §7). Without user identity the plugin cannot safely act on
"the current user", so the patches reach into the host's authenticated
routes instead of staying inside the plugin sandbox.

v0.2.0 removes those patches by relying on a small upstream change: the
host signs each proxied request with an HMAC of the user identity, and
provisions the plugin with a per-plugin verification key. The full
proposal lives in
[`docs/internal/upstream-rfc-issue.md`](internal/upstream-rfc-issue.md).

## Subprocess model

The host's plugin-process-manager spawns the plugin entry as:

```
node dist/server.js
  cwd:  ~/.claude-code-ui/plugins/account
  env:  PATH, HOME, NODE_ENV, PLUGIN_NAME, PLUGIN_IDENTITY_KEY
  stdio: pipe (host reads readiness JSON from stdout)
```

The plugin must print `{"ready":true,"port":<n>}` to stdout within 10
seconds of being spawned. After that, the host proxies HTTP traffic from
clients at `/api/plugins/account/rpc/*` to `127.0.0.1:<n>/*`. The plugin
binds only to loopback; the host is the only legitimate caller.

`SIGTERM` is the graceful-shutdown signal; the host force-kills after 5
seconds. We close the SQLite handle on the way out and never linger
beyond ~4.5s.

## Identity verification

The host adds three headers to every proxied request when `req.user` is
authenticated:

| Header                       | Value                                |
| ---------------------------- | ------------------------------------ |
| `X-Plugin-User-Payload`      | base64(JSON `{userId, username, iat}`) |
| `X-Plugin-User-Signature`    | `sha256=<hex digest>`                |
| `X-Plugin-User-Algorithm`    | `sha256`                             |

The HMAC key is per-plugin:

```
pluginKey = HMAC-SHA256(JWT_SECRET, "plugin:" + pluginName)
```

…delivered to us as `PLUGIN_IDENTITY_KEY` (hex). `JWT_SECRET` itself
never leaves the host. Renaming the plugin or rotating `JWT_SECRET`
invalidates old signatures automatically.

`src/server/auth.ts` enforces, in order:

1. All three headers present; algorithm pinned to `sha256`.
2. Payload base64 length capped at **1024** chars (the legitimate
   payload is ~120 bytes).
3. Signature header parses as `sha256=<even-length lowercase hex>`.
4. Plugin key decodes from hex to a non-empty buffer.
5. Recomputed HMAC matches the supplied signature, **only after** an
   explicit length check, then via `crypto.timingSafeEqual` on
   equal-length buffers.
6. Payload `JSON.parse` wrapped in try/catch.
7. Required fields (`userId`, `username`, `iat`) present and well-typed.
8. Replay window: `now - iat <= 60` (stale) **and** `iat - now <= 5`
   (future-skew).

Any failure returns `null` without throwing. The HTTP layer maps `null`
to **HTTP 401**.

The verifier is unit-tested for: roundtrip, tampered payload, tampered
signature, missing headers, wrong algorithm, stale `iat`, future `iat`,
oversized payload, malformed JSON, equal-length-wrong signatures,
unequal-length signatures, and per-plugin key isolation.

## Endpoints

All endpoints live under `/api/plugins/account/rpc/*` from the client's
perspective and arrive at the plugin without that prefix. Auth is
required on every endpoint except `/health`.

| Method | Path                | Purpose                              |
| ------ | ------------------- | ------------------------------------ |
| GET    | `/health`           | Liveness; no auth.                   |
| GET    | `/me`               | Current user `{id, username}`.       |
| POST   | `/change-password`  | Verify current password (bcrypt) and update. Rate-limited 5/15min. |
| POST   | `/change-username`  | Same auth gate; updates `username`. Rate-limited. |
| GET    | `/sessions`         | Placeholder (host owns sessions).    |

Validation mirrors v0.1.x: usernames `[a-zA-Z0-9_]{3,32}`, passwords
≥ 8 chars and not equal to the current one. `SQLITE_CONSTRAINT_UNIQUE`
on username updates maps to **HTTP 409**.

## Database access

`src/server/db.ts` opens `~/.claude-code-ui/auth.db` via
`better-sqlite3` with `fileMustExist: true` and WAL journal mode (the
host already keeps the DB in WAL; we cooperate). All three statements
(`SELECT users`, `UPDATE password_hash`, `UPDATE username`) are
prepared once at startup and use parameterized values exclusively.

`bcrypt` and `better-sqlite3` are loaded via dynamic `import()` so the
unit tests — which never touch the DB — don't need them on disk. The
host installer runs `npm install --ignore-scripts` followed by
`npm run build` when the plugin is installed, so both native modules
land under the plugin's own `node_modules` at runtime.

## Rate limiting

`src/server/rateLimit.ts` is a tiny in-process Map-keyed limiter with
the v0.1.x policy (5 attempts / 15 minutes), keyed by
`<remoteAddress>:<userId>`. We deliberately do not pull in
`express-rate-limit`; the plugin server has no Express dependency.

## Frontend

The existing `src/index.ts` (vanilla DOM, ~11 KB built) is kept as the
`entry`. Its API client (`src/api.ts`) currently calls the v0.1.x host
routes (`/api/auth/account/*`); when v0.2.0 ships end-to-end, the
client will be retargeted at `/api/plugins/account/rpc/*` (a one-line
change). Keeping the v0.1.x targets in this branch lets the existing 13
api.test.ts cases continue to pass unchanged, which is the cheapest
proof that the experimental branch hasn't regressed the released
plugin's behavior.

## Environment variables consumed

| Var                    | Source                              | Required |
| ---------------------- | ----------------------------------- | -------- |
| `PLUGIN_NAME`          | host (always set to `"account"`)    | no (defaults to `account`) |
| `PLUGIN_IDENTITY_KEY`  | host (RFC, hex of derived HMAC key) | yes — without it, all protected routes return 401 |
| `HOME`                 | host                                | yes (resolves DB path) |

## Upstream dependencies (blockers)

Until the upstream RFC is accepted and shipped, **this build will start
but every authenticated endpoint will return 401**, because the host
neither sets `PLUGIN_IDENTITY_KEY` nor signs the identity headers. The
plugin logs this fact at startup and surfaces it via 401s rather than
crashing, which is intentional: it makes the missing-host-feature
visible from the UI.

The exact host-side changes required are enumerated in
[`docs/internal/upstream-rfc-issue.md`](internal/upstream-rfc-issue.md)
§"Implementation plan".

## Non-goals for v0.2.0

- WebSocket support for the plugin (not currently needed; the RFC
  covers it for future plugins).
- Permission enforcement beyond declaring `"user.identity"` — the host
  doesn't enforce permissions yet (RFC open question 7).
- Self-installation or auto-discovery; this remains a host-installed
  plugin.
- Migrating frontend clients to the new RPC paths (one-line follow-up
  once the host signs identity).
